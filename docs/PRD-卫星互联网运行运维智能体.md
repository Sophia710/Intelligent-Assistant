## 一、产品概述

### 1.1 产品定位

**卫星互联网运行运维智能体** 是一款面向卫星互联网测试与运维领域的 AI 智能助手平台。产品以对话式交互为核心，整合智能体市场（Agent Marketplace）、知识库管理（Knowledge Base）、文档管理（Document Management）三大功能模块，为卫星通信领域的测试工程师、运维人员和技术决策者提供一站式的智能化工作台。

### 1.2 目标用户

| 用户角色 | 描述 | 核心诉求 |
|---------|------|---------|
| **测试工程师** | 负责卫星终端、链路、载荷的测试执行 | 快速生成测试用例、自动化脚本、仿真数据分析 |
| **运维工程师** | 负责在轨卫星的日常监控与故障处理 | 实时遥测分析、故障预测、运维建议 |
| **技术管理者** | 负责测试规划与技术决策支持 | 测试报告归档、知识沉淀、团队协作 |
| **系统管理员** | 负责平台配置与知识库维护 | 知识库管理、文档上传、权限控制 |

### 1.3 核心价值主张

- **AI 驱动的对话式交互**：通过自然语言完成测试任务规划、代码生成、数据分析等复杂操作
- **专业化智能体生态**：覆盖终端测试、星地网络、卫星载荷、全链路验收四大测试领域
- **企业级知识管理**：基于 RAG 技术的知识库体系，支撑协议规范、测试报告、系统文档的结构化管理
- **端到端工作流闭环**：从新建对话 → 选择/创建智能体 → 对话交互 → 知识检索 → 文档引用的完整链路

### 2.6 技能中心 (06) — 页面原型与功能

> **目标**：扩展智能体能力边界。用户可在技能商店浏览/添加系统预置技能，并自定义专属技能，统一收纳到"我的技能"中以备在对话时按需调用。

#### 2.6.1 页面结构

技能中心采用 **双 Tab + 单页布局**：

| 区域 | 内容 |
|------|------|
| Header | 两个主 Tab：`技能商店`（默认） / `我的技能`（含未读数徽标） |
| Tab=技能商店 | 顶部操作条（购买记录 + 创建新技能）→ 分类 Chips（11 个）→ 标签筛选（6 个）→ 行业技能包（2×2 大卡）→ 数据集（3 列卡片） |
| Tab=我的技能 | 顶部操作条（开通商店 + 购买记录 + 搜索框）→ 已添加技能网格（响应式 1/2/3 列） |
| 全局 | 创建自定义技能模态框（弹层） |

#### 2.6.2 路由

| 路由 | 页面 | 说明 |
|------|------|------|
| `/skills` | `SkillsCenterPage` | 技能中心主入口 |
| `/skills?tab=store` | 同上 | 显式切换到技能商店 |
| `/skills?tab=mine` | 同上 | 显式切换到我的技能 |

URL Search Params 控制 Tab 状态，支持刷新保留 + 浏览器前进后退 + 分享。

#### 2.6.3 数据模型

| 模型 | 字段 | 存储位置 |
|------|------|----------|
| 内置技能包 (SkillPack) | id, name, summary, icon, iconBg, category, usage, subSkills[] | `src/data/seedSkills.js`（前端静态） |
| 内置数据集 (Dataset) | id, name, creator, tag, tagType, summary, icon, iconBg, category | 同上 |
| 内置单技能 (Skill) | id, name, description, icon, iconBg, category | 同上 |
| 用户已添加 ID 集合 (UserAddedIds) | string[]（skillId 集合） | `localStorage['user:skills:added:v1']` |
| 自定义技能 (CustomSkill) | id, name, description, icon, iconBg, triggerCommand, promptTemplate, createdAt | `localStorage['user:skills:customSkills:v1']` |

#### 2.6.4 核心交互

| 编号 | 交互 | 行为 |
|------|------|------|
| INT-S01 | 切换 Tab | 200ms `tab-fade-in` 淡入；URL 同步 `?tab=...` |
| INT-S02 | 添加内置技能 | 写入 localStorage + 按钮变"已添加" + Toast 提示 |
| INT-S03 | 移除已添加技能 | 卡片菜单 → 移除 → Toast 提示 + 卡片淡出 |
| INT-S04 | 创建自定义技能 | 模态框 → 表单校验 → 写入 localStorage → 自动切到"我的技能" |
| INT-S05 | 搜索我的技能 | 客户端实时过滤（按名称 + 描述） |
| INT-S06 | 分类筛选 | 单选 Chips，立即过滤技能包 + 数据集 |
| INT-S07 | 标签筛选 | 多选 Toggle，互不互斥 |
| INT-S08 | 立即使用 | 携带 `?skill=...` 跳到新建对话页 |
| INT-S09 | 卡片 hover | 浮起 + 边框主题色 + 阴影加深 |

#### 2.6.5 视觉规范

- **卡片背景**：`--color-surface-container-lowest`
- **卡片边框**：`--color-outline-variant`，hover 时混入 40% primary
- **分类 Chips 激活**：`--color-primary` 底 + `--color-on-primary` 字
- **添加按钮（未添加）**：`--color-on-surface` 底 + `--color-surface` 字
- **添加按钮（已添加）**：透明 + outline + 次级文字
- **图标背景**：使用渐变色（linear-gradient），不随主题变化（保持品牌识别）
- **响应式断点**：`md` 2 列 / `xl` 3 列
- **过渡动画**：`tab-fade-in`（200ms）、`skill-card-hover`（200ms 浮起）、`toast-pop`（220ms 弹入）

##### 4.3.7.1 输出阶段状态机

每条 AI 回复在生命周期中依次经历以下阶段：

```
loading ──5s──▶ thinking ──输出完毕──▶ answer ──输出完毕──▶ done
                  │                      │
                  └──── 停止 ────────────┴────▶ stopped
```

| 阶段 | 触发条件 | 持续 / 终止条件 | UI 表现 |
|------|---------|----------------|---------|
| `loading` | 点击「重新生成」 | 固定 5 秒（可由后端握手耗时覆盖） | 旋转圆环 + 提示语 + 跳动圆点 |
| `thinking` | `loading` 结束 | 流式输出思考内容，输出完后切到 `answer` | 思考过程区强制展开，带流式光标 |
| `answer` | 思考内容已输出完毕 | 流式输出答案内容，输出完后切到 `done` | 答案区带流式光标 |
| `done` | 答案内容全部输出 | 稳定终态 | 思考区自动收起；答案区无光标；可重新生成 |
| `stopped` | 用户在 `thinking` 或 `answer` 阶段点击「停止」 | 稳定终态 | 保留已输出部分 + 「已停止生成」标识 |

##### 4.3.7.2 加载状态（loading）

| 元素 | 规范 |
|------|------|
| 触发 | 用户点击「重新生成」按钮 |
| 时长 | 5 秒（常量，可后续接入真实接口握手耗时） |
| 视觉 | 圆环旋转动画 + 主标题「生成中」 + 副标题「正在调用模型，请稍候...」 + 三个跳动圆点（stagger 0/150/300ms） |
| 是否可中断 | 否（此阶段不响应「停止」按钮） |
| 中断 | 若用户刷新页面 / 切换对话，自动清除该消息的 loading 定时器 |

##### 4.3.7.3 思考过程区（thinking）

| 元素 | 规范 |
|------|------|
| 容器 | 圆角 `rounded-lg` + 左侧 4px 边条 + `mb-3` |
| 头部 | `psychology` 图标 + 「思考过程」文字（`label-md` 12px 中等字重） + 阶段流式光标（仅 `thinking` 阶段显示） |
| 展开 / 收起 | `thinking` 中强制展开；`done` / `stopped` 后默认收起为 2 行；用户可点击头部「展开 / 收起」按钮手动切换 |
| 内容长度 | 思考文案为预置的多行文本（mock），典型 > 10 行；展收框 240px（≈10 行）为上限 |
| 滚动条控制 | 展开后实际高度 ≤ 10 行时**不**显示滚动条且高度自适应；> 10 行时显示细滚动条（`scrollbar-width: thin`） |
| 配色 | `thinking` 阶段：白底 70% 透明 + 紫色左边条 `#5852ff` + 紫文字 + 阴影；`done` / `stopped` 阶段：白底 40% 透明 + 灰左边条 `#c7c4d9` + 灰文字 |
| 数据来源 | 预置常量 `THINKING_TEXT`（卫星互联网测试链路思考分析），重新生成时**必须保留同一份内容**，不允许被清空或覆盖 |
| 复用 | 初始页面加载时也展示一个 mock 思考过程框（默认收起，可手动展开），保证「首次进入」与「重新生成」视觉一致 |

##### 4.3.7.4 答案流式输出（answer）

| 元素 | 规范 |
|------|------|
| 输出速度 | 约 4 字符 / 30ms（前端模拟流式，可由后端 chunk 间隔覆盖） |
| 视觉指示 | 阶段内追加闪烁光标 `BlinkingCaret`（CSS `animate-pulse`） |
| 内容格式 | 支持轻量 Markdown：标题（`#`/`##`/`###`）、段落、有序 / 无序列表、引用块、表格、`**bold**`、代码块、链接、图片 |
| 代码块 | `CodeBlock` 组件：深色头（`#1c1b1b` 背景 + `#313030` 工具条）+ 语言标签 + 「复制代码」按钮 |
| 表格 | 识别表头分隔行（`\|---\|` / `\|:--:\|`），渲染为带斑马纹 hover 的 `<table>` |
| 图片 | `![alt](url)` 末尾追加 image block：全宽圆角 + 居中说明文字 |
| 完成态 | `done` 阶段使用解析器一次性解析完整内容（避免在流式阶段反复解析） |

##### 4.3.7.5 终止与重生成（stop / regenerate）

| 触发 | 行为 |
|------|------|
| 点击「重新生成」 | 仅当消息处于 `done` / `stopped` 时可触发；进入 `loading` 阶段；清理停止信号 |
| 点击输入栏「停止」 | 立即置位 `stopFlagsRef[msgId] = true`；下一次 tick（30ms 内）将 phase 切换为 `stopped`；保留已输出部分 |
| 流式阶段输入栏 | 「发送」按钮自动切换为红色「停止」按钮（`bg-[#ba1a1a]` + `stop_circle` 图标） |
| 停止后状态 | 答案区显示已输出文本 + 「已停止生成（显示部分内容）」灰色提示，无光标 |
| 重新触发 | 用户可继续编辑输入框发送新消息，或再次点击「重新生成」 |

##### 4.3.7.6 自动滚动抑制

| 场景 | 行为 |
|------|------|
| 新消息追加 / AI 输出推进 | 自动滚动到底部（`scrollIntoView({ behavior: 'smooth' })`） |
| 用户点击「重新生成」/「点赞」/「踩」/「复制代码」 | 该次操作触发的 `messages` 变更**不**自动滚动；其他情况恢复 |
| 抑制机制 | `suppressAutoScrollRef` 标志位 + `useEffect` 配合 `preventDefault` / `stopPropagation`；点击处理函数统一经过 `beforeActionButtonClick` 包装 |

| 场景 | 行为 |
|------|------|
| 进入 `/chat/conv-001` 或新建对话 | 消息区不渲染任何 AI 消息气泡 |
| `messages.length === 0` | 渲染空状态占位：圆角图标 + 「在下方输入框中开始第一轮对话」+ 「系统不会在用户输入前发送任何消息」副提示 |
| 用户点击「发送」且首条消息发出 | 走标准 `loading → thinking → answer → done` 阶段机 |

> 副提示文案仅作为 UI 占位引导，**不属于 AI 回复**，因此不违反"无初始消息"原则。

##### 4.3.7.10 AI 回答失败态（Failure State）

**触发条件**：AI 消息的 `status === 'failed'`（可由后端显式标注或前端在 `regenerateMessage` 失败时设置）。

**视觉规范**（与成功态完全区分）：

| 维度 | 成功态 | 失败态 |
|------|-------|-------|
| 头像背景 | `#5852ff`（紫） | `#ba1a1a`（红） |
| 头像图标 | `robot_2` FILL 1 | `smart_toy` |
| 气泡背景 | `#f6f3f2`（浅灰） | `#ffdad6`（红 90 浅） |
| 气泡边框 | `border-[#e5e2e1]/50` | `border-[#ba1a1a]/30` |
| 气泡内文字 | `#1c1b1b` | `#410002` |
| 标签行 | 无 | 「回答失败」红字 + `error` 图标 + 错误原因（10px，右上角截断 + title 提示） |
| 思考区 | 显示 THINKING_TEXT | **不显示** |
| Markdown 渲染区 | 完整渲染 | **不渲染**，改为失败提示正文（`whitespace-pre-wrap`） |
| 操作按钮 | 点赞 / 踩 / 重新生成 | **仅「重试」按钮**（红字 + `refresh` 图标） |
| 反馈提示 | — | 右下角小字「本次回答未计入有效反馈」 |

**字段约定**（失败态消息对象）：

```json
{
  "id": "msg-xxx",
  "role": "assistant",
  "status": "failed",
  "content": "抱歉，我暂时无法回答这个问题……",   // 用户可读的失败提示
  "error": "大模型服务返回 503：上游推理服务暂时不可用",  // 细颗粒度原因
  "retry_content": "## 重试成功的内容...",          // 点击「重试」时驱动流式输出
  "created_at": "..."
}
```

**重试机制**：
1. 用户点击失败气泡内的「重试」按钮 → 触发 `handleRegenerate(messageId)`；
2. `regenerateMessage` 在 hook 层自动清除 `status: undefined`（无需手动 setState）；
3. `handleRegenerate` 优先从 `message.retry_content` 读取新内容，否则回退到 `content`；
4. 进入标准 `loading → thinking → answer → done` 阶段机，气泡自动从失败态切换为成功态。

##### 4.3.7.11 失败示例 Mock 对话（demo）

| conversationId | 内容 | 用途 |
|---------------|------|------|

## 七、用户故事 (User Stories)

### 7.1 Epic 1：对话交互

**US-CHAT-01**: 作为测试工程师，我希望能够通过自然语言与 AI 对话，以便快速获取测试方案建议。
- **验收标准**: 输入问题后 3 秒内收到 AI 回复；回复内容与问题相关

**US-CHAT-02**: 作为测试工程师，我希望 AI 能直接输出可执行的 Python 代码并支持一键复制，以便我无需手动编写。
- **验收标准**: 代码块有语法高亮；复制按钮可用且准确复制全部代码

**US-CHAT-03**: 作为测试工程师，当 AI 回复不满意时，我希望可以请求重新生成，以便获得更好的答案。
- **验收标准**: 点击「重新生成」后原回复被替换为新回复；加载中有明确指示

**US-CHAT-04**: 作为测试工程师，我希望看到 AI 的思考过程，以便了解答案的推导逻辑并建立信任。
- **验收标准**: 思考过程区在 AI 思考中强制展开；思考完成后自动收起为 2 行预览；用户可手动展开查看完整内容

**US-CHAT-05**: 作为测试工程师，当思考内容超过 10 行时，我希望只显示一个固定高度的滚动区而不要撑高整个对话流，以便保持界面整洁。
- **验收标准**: 展开后最大高度 240px；超出时显示细垂直滚动条；未超出时不显示滚动条且高度自适应

**US-CHAT-06**: 作为测试工程师，我希望在 AI 长时间生成过程中随时点击「停止」中断当前回答，以便节省等待时间。
- **验收标准**: 流式阶段输入栏出现红色「停止」按钮；点击后 30ms 内停止生成；已输出部分保留并以「已停止生成」标识提示

**US-CHAT-07**: 作为测试工程师，我希望 AI 回复自动支持代码块、表格、列表、标题等富媒体结构，以便提升长答案的可读性。
- **验收标准**: Markdown 语法（`#`/`##`/`###` / `-` / `1.` / `>` / `|...|` / ` ``` ` / `**` / `![]()`）均能正确渲染

**US-CHAT-08**: 作为新用户，我希望首次进入对话页就能看到示例回答与思考过程框，以便理解产品能力并快速上手。
- **验收标准**: 初始加载即展示 mock 用户消息 + mock AI 富媒体回答 + mock 思考过程可展收框；后端无响应时不影响展示

**US-CHAT-09**: 作为新用户，我希望首次进入对话页时**不看到**任何 AI 自动问候或预生成回复，以便清楚「第一轮对话必须由我主动发起」的产品规则。
- **验收标准**: 进入 `/chat/conv-001` 或新建对话时消息区为空 / 仅显示 UI 占位引导（不含 AI 消息气泡）；点击「发送」后才进入 AI 回复流程

**US-CHAT-10**: 作为测试工程师，当 AI 返回失败（如知识库未命中 / 上游服务 503）时，我希望看到明确的失败提示和重试入口，以便快速恢复工作流。
- **验收标准**: 失败时气泡为红底配色 + `error` 图标 + 「回答失败」标签 + 错误原因；操作区**仅**显示「重试」按钮（隐藏点赞 / 踩 / 重新生成）；点击「重试」后走标准 `loading → thinking → answer → done` 阶段机并展示成功内容

**US-CHAT-11**: 作为测试工程师，我希望在侧边栏就能看到历史对话的「成功 / 失败」状态标识，以便快速定位失败示例并参照改进。
- **验收标准**: Sidebar 列表项前缀图标与末尾小标签按状态区分（成功：紫色 / 失败：红色 `error` + 「失败」小标签 / 空：新对话图标）
